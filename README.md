# 要件定義書：北海道ツーリングルート記録アプリ

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | 北海道ツーリングルート記録アプリ（仮称） |
| 開発者 | 悠斗 |
| 技術スタック | FastAPI + PostgreSQL（→ Phase 3でGo移植予定） |
| 開発手法 | アジャイル（1週間スプリント） |

---

## 2. ユーザーストーリー

**ユーザー：** 自分（北海道在住・Majesty Cで週1日帰りツーリング）

**課題：** 次のツーリングルートを計画するとき、以下の情報がまとまっていない
- 過去に走ったルート沿いのガソリンスタンドの場所・間隔
- トイレが使えるスポットの位置
- ソフトクリームが買える店の場所
- その他、自分が重要だと思ったスポット

**ゴール：** 過去のツーリング記録からスポットをカテゴリ別に振り返り、次のルートを効率的に計画できる

---

## 3. 機能要件

### 3.1 ルート管理（CRUD）

| 機能 | 説明 |
|------|------|
| ルート登録 | ルート名（例：「女満別〜網走」）、出発地、目的地、走行日、メモを登録 |
| ルート一覧 | 登録したルートを一覧表示（新しい順） |
| ルート詳細 | 特定ルートの詳細と紐づくスポットを表示 |
| ルート編集 | 登録済みルート情報の更新 |
| ルート削除 | ルートの削除（紐づくスポットも削除） |

### 3.2 スポット管理（CRUD）

| 機能 | 説明 |
|------|------|
| スポット登録 | ルートに紐づけてスポットを登録（名前、カテゴリ、メモ） |
| スポット一覧 | ルート内のスポットをカテゴリ別に表示 |
| スポット編集 | スポット情報の更新 |
| スポット削除 | スポットの削除 |

### 3.3 カテゴリ管理

| 機能 | 説明 |
|------|------|
| カテゴリ登録 | 自由にカテゴリを追加できる（例：ガソリンスタンド、トイレ、ソフトクリーム、コンビニ、絶景ポイント） |
| カテゴリ一覧 | 登録済みカテゴリの一覧 |
| カテゴリ編集 | カテゴリ名の変更 |
| カテゴリ削除 | カテゴリの削除 |

**初期カテゴリ（seed data）：**
- ガソリンスタンド
- トイレ
- ソフトクリーム

---

## 4. データモデル（ER設計）

### routes（ルート）

| カラム | 型 | 説明 |
|--------|------|------|
| id | UUID | 主キー |
| name | VARCHAR(100) | ルート名（例：「女満別〜網走」） |
| start_point | VARCHAR(100) | 出発地 |
| end_point | VARCHAR(100) | 目的地 |
| date | DATE | 走行日 |
| memo | TEXT | メモ（任意） |
| created_at | TIMESTAMP | 作成日時 |
| updated_at | TIMESTAMP | 更新日時 |

### spots（スポット）

| カラム | 型 | 説明 |
|--------|------|------|
| id | UUID | 主キー |
| route_id | UUID | 外部キー → routes.id |
| category_id | UUID | 外部キー → categories.id |
| name | VARCHAR(100) | スポット名（例：「セイコーマート網走店」） |
| memo | TEXT | メモ（任意、例：「24時間営業」「ソフトクリーム美味い」） |
| created_at | TIMESTAMP | 作成日時 |

### categories（カテゴリ）

| カラム | 型 | 説明 |
|--------|------|------|
| id | UUID | 主キー |
| name | VARCHAR(50) | カテゴリ名 |

**リレーション：**
- routes 1 : N spots（1ルートに複数スポット）
- categories 1 : N spots（1カテゴリに複数スポット）

## 5. APIエンドポイント設計

### ルート

| メソッド | パス | 説明 |
|----------|------|------|
| POST | /routes | ルート新規登録 |
| GET | /routes | ルート一覧取得 |
| GET | /routes/{id} | ルート詳細取得（スポット含む） |
| PUT | /routes/{id} | ルート更新 |
| DELETE | /routes/{id} | ルート削除 |

### スポット

| メソッド | パス | 説明 |
|----------|------|------|
| POST | /routes/{route_id}/spots | スポット登録 |
| GET | /routes/{route_id}/spots | ルート内スポット一覧 |
| PUT | /spots/{id} | スポット更新 |
| DELETE | /spots/{id} | スポット削除 |

### カテゴリ

| メソッド | パス | 説明 |
|----------|------|------|
| POST | /categories | カテゴリ登録 |
| GET | /categories | カテゴリ一覧 |
| PUT | /categories/{id} | カテゴリ更新 |
| DELETE | /categories/{id} | カテゴリ削除 |

## 7. 開発フェーズ

### Phase 1：API基盤（1〜2週間）
- FastAPIプロジェクト作成
- PostgreSQL接続（SQLAlchemy）
- routes / spots / categories のCRUD API実装
- Swagger UIで動作確認
- **完了条件：** 全エンドポイントがSwagger UIで正常動作する

### Phase 2：フロント + デプロイ（1週間）
- 簡易フロント（HTML/JS）でルート・スポットの登録・閲覧
- Renderにデプロイ
- GitHubにコード公開
- **完了条件：** URLを共有して動作確認できる

### Phase 3：機能拡張（継続）
- 地図表示（Leaflet連携）
- 画像アップロード
- 検索・絞り込み（カテゴリ別、季節別）
- Goでの書き直し

---

## 8. 面接で語れるポイント（想定）

- **なぜ作ったか：** 北海道でバイクに乗っていて、ガソリンスタンドやトイレの間隔がわからず困った実体験
- **技術選定の理由：** FastAPIは軽量で学習コストが低く、Swagger UIで即座にAPI動作確認できる
- **DB設計：** routes - spots - categories の1対多リレーション設計
- **アジャイル開発：** 1週間スプリントで段階的に機能追加
- **Go移植（Phase 3）：** 同じ仕様を2言語で実装し、開発体験の違いを比較
